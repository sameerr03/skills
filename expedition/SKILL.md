# Expedition

## Purpose

Execute an accepted outcome-complete and dependency-complete Atlas map. The orchestrator owns the epic, implementation planning for each leaf, dependency graph, dedicated feature branch, stack, shared environments, cross-PR coherence, code review, and acceptance. Each persistent worker executes exactly one orchestrator-planned implementation issue, runs its self-review loop, and owns that PR.

Atlas explicitly defines consequential technical changes. The orchestrator resolves ordinary internal implementation choices from the live code and the user's intent before instructing a worker. If execution requires an unplanned schema, migration, authentication, public API, persistent-state, service-boundary, dependency, provider, destructive-data, cross-leaf, product, security, compatibility, or UX decision, stop and return to the user or Atlas. Expedition executes the route; it does not silently redesign it.

## Stack Topology

Create exactly one dedicated feature branch for the whole Expedition from the exact original target head. Planning, Atlas, scratch, and validation branches are never implementation roots or alternate merge targets. Record the original target and canonical feature branch before launching workers.

The published implementation topology is always one linear stack:

```text
original target <- expedition feature branch <- PR 1 <- PR 2 <- ... <- final PR
```

The Atlas dependency graph controls eligibility and safe parallel execution only; it is not permanent Git ancestry. Parallel workers may start from the same accepted base, but their branches remain provisional until serialized. Maintain one canonical stack ledger containing the ordered leaf, branch, exact head, and immediate PR base. As parallel leaves finish, restack them in that order onto the current canonical head and update each PR base before accepting them or using them as a downstream base. A completed provisional branch never becomes a second canonical target. Do not create a synthetic integration branch or merge commits to combine parallel leaves.

The first leaf PR targets the Expedition feature branch. Every later PR targets the immediately preceding serialized leaf branch. Therefore the final PR head contains the complete system and is the sole combined validation surface. The feature branch remains the eventual single path back to the original target.

During Expedition, never merge a PR, enable auto-merge, merge or cherry-pick an open leaf into its PR base, or push a base branch so that it contains its open PR head. GitHub treats that as a merge even when performed locally. Combined testing uses the latest serialized leaf head, never an advanced base branch.

## Ownership and Observation

- **Orchestrator owns:** the epic and stack, leaf planning, worker instructions, review of delivered heads, shared environments, UI/UX acceptance through Computer Use, cross-leaf decisions, and the final accept-or-return decision. It sends findings to the worker; it does not edit, commit, push, or update the worker's PR.
- **Persistent worker owns:** its leaf worktree, branch, code, validation, `self-review-loop`, commits, pushes, PR state, and one complete `automated-pr-review-loop` phase. Inside that phase it adjudicates bot findings, applies valid fixes, and requests re-review until terminal. After terminal completion, the automated phase is permanently done for that leaf even if later orchestrator feedback produces another commit.
- **Self-review subagent owns:** one independent read-only review. It never edits the shared worktree or takes ownership from the worker.

The orchestrator remains responsible for the Expedition while the worker runs. It observes through occasional read-only status checks and waits without taking over the worker's work. It steers only under the intervention conditions documented in **Orchestrator review** below. When the worker reports a reviewed head, the orchestrator either sends consolidated findings to that same worker or tells it to begin the automated gate. After the worker reports that gate complete, the orchestrator performs final verification and acceptance.

## Execution Loop

Use this acceptance order for every leaf:

```text
orchestrator: plan and launch
worker: implement -> validate -> self-review-loop -> publish PR
orchestrator: review code -> perform required UI/UX acceptance
worker: run one complete automated-pr-review-loop phase -> report terminal disposition
orchestrator: final verification -> later fixes never restart automated review
```

1. **Load the map and goal.** Read the epic's `## Expedition Goal`, descendants, parent/sub-issue relationships, blocked-by edges, linked PRs, repository instructions, and current branch state. When the user explicitly invokes Expedition, initialize the thread's active goal from that exact objective and its completion conditions when a goal mechanism is available; do not silently reinterpret or broaden it. Verify the graph is acyclic and each executable leaf is a one-PR outcome contract; do not expect Atlas to prescribe ordinary implementation internals. Identify the original target branch and exact head, then create and push one clearly named Expedition feature branch from that head before launching workers. If the proposed feature branch already exists, verify its ownership, ancestry, and exact head instead of overwriting or repurposing it. Enable `automated-pr-review-loop` for the Expedition by default and record any explicit user opt-out before publishing PRs.
2. **Choose work.** Native dependencies define execution eligibility, not final PR ancestry. Parallelize safe independent leaves when useful. Before parallel launch, record their eventual serial order and mark every non-canonical starting base as provisional. As leaves finish, serialize them in that order: restack each leaf onto the current canonical head, retarget its PR to that immediate predecessor, verify ancestry, then advance the canonical head. A downstream leaf is stack-ready only when that one serialized head contains every required accepted prerequisite. Never launch or retarget downstream work from a merely completed provisional branch.
3. **Plan the leaf.** Before launching a worker, inspect the live issue, accepted ancestors, relevant code, tests, and repository patterns. Produce the smallest concrete implementation plan that satisfies the issue: identify existing code to reuse, intended data and control flow, necessary code surfaces, direct failure cases, validation, exclusions, and PR base. For each planned new or changed function or meaningful behavior boundary, state its behavioral contract and the exact unit tests for its success cases and direct failures, including retry or idempotency only when applicable. Prefer tests of observable behavior; do not invent private helpers or brittle one-test-per-function structure merely to make this mapping. Preserve every consequential decision in the Atlas leaf. If the live code requires a new consequential decision or materially different boundary, stop for the user or Atlas instead of delegating the ambiguity to a worker.
4. **Launch one persistent worker.** Create a separate Codex task in a new worktree for the leaf. Use `gpt-5.6-sol` with high reasoning for implementation workers unless the user explicitly requests another available model. Give it the issue contract, the orchestrator's implementation plan, original target, canonical Expedition feature branch, exact starting base, current serialized predecessor, branch and PR base, exclusions, repository instructions, validation, and ownership boundaries. Include these instructions verbatim: "Follow the orchestrator's implementation plan and implement the smallest readable diff that satisfies the GitHub issue. Reuse existing code and do not add abstractions, recovery systems, edge-case handling, or tests unless required by the issue, the orchestrator's plan, or a demonstrated regression. Write and run the orchestrator-specified behavior tests before writing production code, confirm that they fail for the intended missing behavior, then implement the smallest change that makes them pass. Test the specified behavior and direct failure cases only. Do not make product, architecture, schema, migration, authentication, public-API, persistent-state, dependency, provider, or cross-leaf decisions. If code evidence contradicts the plan or requires such a decision, stop before expanding the diff and report the evidence, smallest viable options, and your recommendation to the orchestrator. Before publishing, report production and test lines changed and simplify anything not directly tied to the issue." For the first leaf, set the PR base to the Expedition feature branch. For every later leaf, set it to the immediately preceding serialized leaf; a provisional parallel base must be restacked before acceptance. Reuse that same task for every clarification, restack, and fix round.
5. **Resolve worker escalations.** A worker escalation is not a completed delivery. Inspect the reported evidence and only the code needed to resolve it. Decide ordinary issue-bounded implementation questions and send one consolidated clarification to the same worker. If the choice changes a consequential Atlas decision, issue boundary, dependency, or user intent, stop for the user or Atlas. Never tell the worker to choose silently, create a replacement worker, or continue speculatively.
6. **Worker delivery.** The worker implements only the orchestrator-planned issue, runs relevant validation, checks its diff against the worker instruction, and reports production and test lines changed. It then runs `self-review-loop` with an independent read-only subagent, makes only the smallest validated fixes, and reruns affected validation before committing, pushing, and creating or updating a PR whose body uses `Closes #N` for that leaf. When automated PR review is enabled, publish the PR as ready for review; when the user explicitly opts out, publish it as a draft. Verify the PR's base branch and state after creation. Do not run `automated-pr-review-loop` yet. Return the exact head SHA, changed surface, validation and self-review evidence, assumptions, and residual gaps to the orchestrator.
7. **Orchestrator review.** Wait until the worker has completed implementation, validation, self-review, commit, push, and PR delivery before inspecting its code or sending review guidance, unless the worker explicitly escalates or an existing intervention condition applies. While it is active, use occasional read-only status checks only to detect unsafe actions, clear scope runaway, a genuine stall, a rabbit hole, an infinite review loop, or a material assumption that contradicts the issue, orchestrator plan, repository evidence, or an already-settled decision; otherwise let it work without proactive steering. After delivery, independently verify the exact head, complete diff, issue and orchestrator-plan compliance, test evidence, code simplicity, integration with accepted ancestors, and stack ancestry. Batch validated findings back to the same worker. The worker applies them, reruns affected validation and `self-review-loop`, updates the PR, and returns a new exact head before another orchestrator review.
8. **UI/UX acceptance.** For user-visible changes, the orchestrator owns the running environment and performs design and interaction review through Computer Use against the actual app after the code review is clean. Check relevant desktop and narrow widths, themes, keyboard/focus behavior, overflow, states, workflows, visual hierarchy, and coherence with the integrated product. Return findings to the same worker, then repeat affected validation, `self-review-loop`, orchestrator review, and UI/UX acceptance until the exact head is accepted. Skip this step only when the leaf has no user-visible behavior requiring live acceptance.
9. **Automated PR review gate.** Start this phase exactly once and only after the orchestrator has accepted the code review and any required UI/UX acceptance for the current exact head. The same worker owns the complete `automated-pr-review-loop`: trigger the enabled reviewers, handle rate limits, adjudicate findings, apply only valid fixes, validate and self-review those fixes, and request re-review as needed until the skill reaches terminal quiescence. Then record `automated_review_complete` for the leaf and report the final reviewed head and disposition. This terminal marker remains valid for later corrective commits. Reopen automated review only in the rare case where the orchestrator explicitly determines that a later change is so significant that the prior automated review no longer meaningfully covers the implementation—for example, a new schema/migration, a redesigned authentication or production-write boundary, a changed public API contract, or a substantial rewrite of the reviewed core. Merely touching consequential data, scoring, lineage, or production code is not enough. Ordinary bug fixes, tests, output wording, restacks with patch equivalence, and narrow integration corrections never restart it. When the user opted out, skip the phase and keep the PR in draft state.
10. **Accept and advance.** After the worker reports `automated_review_complete`, the orchestrator performs final read-only verification of the reported head, disposition, PR base, and expected state. If that verification or later integrated acceptance finds another code issue, send it to the same worker; the worker may fix, validate, run `self-review-loop`, commit, and push without invoking automated review again. Only a rare change that makes the prior automated review materially obsolete uses a new automated phase, and the orchestrator must explicitly say so before the worker starts it. Verify ordinary corrected heads directly, then accept the leaf when its PR targets the immediately preceding serialized leaf and contains the complete accepted prefix. Record the exact head in the canonical stack ledger and advance the one canonical head. Do not close or merge the PR.
11. **Serialize and propagate.** When parallel implementations finish, keep them provisional and serialize them in the recorded order before acceptance. Each affected worker restacks its leaf patch onto the preceding serialized head, verifies the base-to-head diff still represents only its issue, updates the PR base, reruns affected validation and review gates, and reports the new exact head. After any accepted head changes, mark every later descendant stale until its worker restacks it onto the new canonical prefix. Never converge branches by merging them into a shared base.
12. **Repeat.** Continue until every leaf has an accepted PR in one linear chain and the final PR head contains the complete system. Run final integrated acceptance directly from that final head. Report the eventual user-authorized promotion order: merge PR 1 into the Expedition feature branch, retarget and merge each following PR into that same feature branch in serial order, then merge the feature branch into the original target. Perform none of those merges during Expedition without explicit authorization.

## Authority and Safety

- Workers do not redesign their leaf, resolve consequential ambiguities, modify other leaves, manage shared deployments, perform destructive data actions, merge PRs, or replace orchestrator planning and acceptance. Incidental coding choices must remain within the orchestrator's plan.
- The orchestrator owns leaf implementation plans, final code review, shared dev servers, deployments, migrations, data-safety gates, combined-branch testing, and cross-PR decisions, but never implements leaf fixes or operates the worker's branch or PR. Require explicit user approval for destructive or production actions.
- Enable automated PR review by default. Publish every PR covered by that policy as ready for review, including any eventual feature-branch PR to the original target. Only publish or keep a PR as draft when the user explicitly opts that PR or Expedition out of automated review.
- Never target the first leaf PR at the original target branch. The dedicated Expedition feature branch is the required base and eventual integration boundary.
- Never create or use a second integration branch. Never merge accepted leaves into a base for testing; test the latest serialized leaf head instead.
- Never run `gh pr merge`, enable auto-merge, or push a PR head into its base during Expedition. A user-authorized promotion turn is separate from implementation and must follow the reported serial order.
- Do not create replacement workers for review fixes. Preserve task context by steering the original persistent worker.
- Do not inspect or steer an incomplete worker implementation merely to optimize its approach. Intervene before delivery only for an explicit worker escalation, safety, unauthorized actions, obvious scope escalation, lack of meaningful progress, runaway looping, or a material incorrect assumption visible in the worker's updates.
- Automated review is a default-enabled, worker-owned phase that starts after orchestrator review and UI/UX acceptance. The worker may iterate within that phase until terminal. Afterward, later commits do not restart it unless the orchestrator explicitly declares a rare, substantial redesign that makes the completed review materially obsolete. Important bug fixes and changes in consequential code still use normal validation, self-review, and orchestrator verification without another bot loop. Disable automated review only for an explicit user opt-out.
- Never merge PRs unless the user explicitly authorizes merging. Mark the active goal complete only after every leaf has an accepted exact head and required integrated acceptance is complete. Finish with the epic, PR stack, exact accepted heads, validation dispositions, unresolved blockers, and residual risks.
